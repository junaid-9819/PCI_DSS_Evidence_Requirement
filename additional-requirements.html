The Additional Requirements domain covers specific PCI DSS requirements for multi-tenant service providers, entities using SSL/early TLS for card-present POS POI terminal connections, and other specialized scenarios as outlined in Appendices A1 and A2. These requirements are applicable only to certain entities based on their operational context. The evidence below is consolidated for functional stakeholders (e.g., IT, security, compliance teams), using practical, real-world artifacts aligned with PCI DSS v4.0.1 requirements.

Evidence Requirement PCI DSS Requirement(s) Description and Practical Examples
Multi-Tenant Service Provider Isolation Controls A1.1.1, A1.1.2, A1.1.3, A1.1.4 For multi-tenant service providers, documentation and configurations demonstrating logical and physical isolation of each customer's environment, including CHD storage and processing. Example: A PDF report detailing virtual private cloud (VPC) configurations (e.g., AWS VPC with dedicated subnets) for customer environments, plus access control lists (ACLs) ensuring no cross-tenant access to CHD.
Multi-Tenant Service Provider Access Controls A1.2.1, A1.2.2 Evidence of role-based access controls restricting customer access to their own environments and preventing unauthorized access to other tenants' data. Example: Screenshots from an identity management system (e.g., Okta) showing role-based access for tenant admins, with logs confirming no unauthorized access attempts across tenant boundaries.
Multi-Tenant Penetration Testing Records A1.1.4 Penetration test reports for multi-tenant service providers, conducted annually on customer-subscribed infrastructure, verifying isolation controls and remediating vulnerabilities. Example: A PDF penetration test report from a QSA (e.g., Trustwave) testing tenant isolation in a multi-tenant environment, with findings (e.g., weak API authentication) and evidence of remediation (e.g., updated API security).
SSL/Early TLS Transition Plan A2.1.1 For entities using SSL/early TLS for card-present POS POI terminal connections, a documented plan to transition to secure protocols (e.g., TLS 1.2 or higher), including timelines and risk mitigations. Example: A Word document titled "TLS Transition Plan" outlining a 6-month migration from TLS 1.0 to TLS 1.3 for POS terminals, with interim mitigations (e.g., network segmentation) and a project timeline.
SSL/Early TLS Secure Configurations A2.1.2 Configurations showing secure protocol usage (e.g., TLS 1.2 or higher) or compensating controls for SSL/early TLS in POS POI environments, with evidence of vulnerability remediation. Example: Screenshots of POS terminal configurations (e.g., Verifone terminal settings) showing TLS 1.2 enabled, plus a vulnerability scan report confirming no exploitable SSL/early TLS weaknesses.
SSL/Early TLS Risk Assessment A2.1.1 A risk assessment for entities using SSL/early TLS, identifying risks, justifying continued use (if applicable), and detailing compensating controls. Example: A PDF risk assessment report listing risks of SSL in POS terminals (e.g., man-in-the-middle attacks), justifying temporary use due to legacy systems, and detailing compensating controls (e.g., IPS monitoring, encrypted VPN).
