The Logging and Monitoring domain ensures that all access to system components and cardholder data in the Cardholder Data Environment (CDE) is logged and monitored to detect and respond to security events. The evidence below is consolidated for functional stakeholders (e.g., IT, security, network teams), using practical, real-world artifacts aligned with PCI DSS v4.0.1 requirements.

Evidence Requirement PCI DSS Requirement(s) Description and Practical Examples
Logging and Monitoring Policy 10.1.1, 10.1.2 A documented policy defining processes for logging and monitoring access to CDE system components and cardholder data, including audit log generation and retention. Example: A PDF policy document titled "Logging and Monitoring Policy" outlining log generation for CDE systems (e.g., servers, databases), retention for at least 12 months, and daily review procedures.
Audit Log Configurations 10.2.1, 10.2.2, 10.2.3, 10.2.4, 10.2.5, 10.2.6, 10.2.7 Configurations showing audit logs enabled on CDE systems, capturing security events (e.g., successful/failed logins, access to cardholder data, system changes) and protected from unauthorized modification. Example: Screenshots from a SIEM (e.g., Splunk) or server (e.g., Windows Event Viewer) showing enabled logging for CDE server access, with access controls restricting log modifications.
Daily Log Review Records 10.4.1, 10.4.2 Records of daily reviews of audit logs for critical security controls (e.g., firewalls, authentication servers), identifying and investigating anomalies. Example: A SIEM report in PDF from Splunk showing daily log reviews for CDE firewall events, with analyst notes on investigated anomalies (e.g., failed login attempts).
Periodic Log Review Records 10.4.3 Records of periodic (e.g., weekly or monthly) log reviews for non-critical CDE systems not subject to daily reviews. Example: A quarterly log review summary in Excel for CDE application servers, documenting reviewed events (e.g., configuration changes) and any follow-up actions.
Time Synchronization Configurations 10.6.1, 10.6.2 Configurations showing Network Time Protocol (NTP) or similar time-synchronization mechanisms enabled on CDE systems to ensure accurate log timestamps. Example: Screenshots of NTP settings on CDE servers (e.g., pointing to time.windows.com) and a router configuration file confirming synchronized time across devices.
Log Retention and Protection Evidence 10.5.1, 10.5.2, 10.5.3, 10.5.4 Evidence of audit log retention for at least 12 months (with 3 months immediately available) and protections against tampering (e.g., write-once storage, access controls). Example: A backup system report showing 12 months of archived logs stored on a WORM drive, plus SIEM access control settings restricting log file modifications.
Security Event Response Procedures 10.7.1, 10.7.2, 10.7.3 Documented procedures for responding to security events identified in logs, including escalation and remediation processes, with evidence of responses to critical events. Example: An SOP in Word detailing escalation steps for SIEM alerts (e.g., unauthorized access), plus a ServiceNow incident ticket documenting a responded-to CDE server breach with remediation details.
Log Collection and Correlation Processes (DESV) A3.3.2 For Designated Entities, documented processes for collecting and correlating logs from critical CDE systems and security functions, with evidence of implementation. Example: A SIEM configuration guide in PDF describing log collection from CDE servers and firewalls, plus a Splunk dashboard screenshot showing correlated events (e.g., login failures across systems).
