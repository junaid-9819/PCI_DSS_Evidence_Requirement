The System Security domain ensures that in-scope system components are securely configured and maintained to protect the Cardholder Data Environment (CDE). The evidence below is consolidated for functional stakeholders (e.g., IT, system administration, security teams), using practical, real-world artifacts aligned with PCI DSS v4.0.1 requirements.

Evidence Requirement PCI DSS Requirement(s) Description and Practical Examples
Secure Configuration Processes 2.1.1, 2.2.1 Documented processes for applying secure configurations to all in-scope system components, aligned with industry-accepted hardening standards (e.g., CIS, NIST). Example: An SOP in Word or SharePoint detailing server hardening procedures, referencing CIS Windows Server 2022 benchmarks, with steps for disabling unnecessary services and applying security patches.
System Configuration Standards 2.2.2, 2.2.3, 2.2.4 Documented standards for secure configurations across system types (e.g., servers, databases, applications), addressing known vulnerabilities and removing unnecessary functionality. Example: A PDF configuration guide listing required settings for Windows servers (e.g., disabled Telnet, enabled TLS 1.3) and databases (e.g., MySQL with restricted user privileges), aligned with CIS benchmarks.
System Configuration Evidence 2.2.2, 2.2.3, 2.2.4 Screenshots or configuration files showing secure settings applied to in-scope systems, with unnecessary services, protocols, and default accounts disabled or removed. Example: Screenshots from a server showing disabled FTP, removed guest accounts, and enabled secure protocols (e.g., SSH with AES-256), plus a configuration file for a database with restricted access.
Non-Console Access Encryption 2.2.5 Configurations demonstrating that all non-console administrative access to CDE systems is encrypted using strong cryptography (e.g., SSH, RDP with TLS). Example: SSH configuration file (e.g., sshd_config) showing AES-256 encryption and RDP settings with TLS 1.3 enabled, verified via screenshots from a CDE server.
Vendor Default Account Management 2.2.3 Evidence that vendor-supplied default accounts, passwords, and settings are changed or disabled on in-scope systems. Example: A change log or screenshot showing renamed or disabled default accounts (e.g., "admin" changed to a unique ID) on a payment application or router, with updated passwords.
Technology Review Records (DESV) A3.2.4 For Designated Entities, annual review records confirming that hardware and software technologies in the CDE continue to meet PCI DSS requirements. Example: A PDF report summarizing annual reviews of server OS (e.g., Windows Server 2022) and firewall firmware, noting compliance with PCI DSS and any updates applied.
