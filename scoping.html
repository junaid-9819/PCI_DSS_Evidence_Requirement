The Scoping domain is the first step in a PCI DSS assessment, as it defines the Cardholder Data Environment (CDE) and in-scope system components, people, and processes. Accurate scoping ensures all subsequent evidence collection aligns with the environment that stores, processes, or transmits cardholder data (CHD) and/or sensitive authentication data (SAD), or could impact its security. The evidence below is consolidated to engage functional stakeholders (e.g., IT, network, compliance teams) efficiently, using practical, real-world artifacts that assessors can validate.

Evidence Requirement PCI DSS Requirement(s) Description and Practical Examples
Scope Confirmation Documentation 12.5.2 A comprehensive document confirming the PCI DSS scope, updated annually (every 6 months for service providers), identifying all locations, flows, and storage of account data (CHD and SAD), and all systems connected to or impacting the CDE (e.g., authentication servers, SIEM). The document details the scoping process.

Example: A Word or PDF document titled "PCI DSS Scope Report" listing payment processing systems (e.g., POS terminals, payment gateways), data storage locations (e.g., databases, backups), and connected systems (e.g., logging servers), with a narrative explaining the scoping methodology (e.g., data discovery scans, IT interviews).
Data-Flow Diagrams 12.5.2 Up-to-date diagrams illustrating CHD and SAD flows, including entry points (e.g., POS, e-commerce), processing systems, storage locations, and transmission to external parties (e.g., acquirers). Diagrams show segmentation boundaries if used.

Example: A Visio or Lucidchart diagram showing data flow from POS terminals to a payment processor via a firewall, with arrows indicating CHD transmission and storage in a database, and segmented non-CDE networks labelled.
System Component Inventory 12.5.2 A current inventory of in-scope system components (e.g., servers, network devices, applications, virtual machines, cloud components) with function, location, and CDE role.

Example: An Excel spreadsheet listing each component (e.g., "Payment Server 01, IP 192.168.1.10, hosts payment application"), including OS, location (e.g., on-premises data center, AWS), and CDE connectivity (e.g., direct, segmented).
Segmentation Validation Records 12.5.2 If segmentation is used, documentation validating its effectiveness, showing out-of-scope systems cannot impact the CDE. Includes network configurations and test results (e.g., penetration testing for DESV).

Example: A report summarizing firewall rules isolating the CDE (e.g., ACLs restricting traffic to specific ports), plus a penetration test report confirming no access from non-CDE systems to the CDE.
Data-Discovery Scan Results 12.5.2, A3.2.1 (DESV) Records of quarterly data-discovery scans to locate all CHD/SAD storage, including unintended locations (e.g., cleartext PAN outside the CDE).

Example: A report from a data-discovery tool (e.g., Spirion, Varonis) listing scanned systems, identified PAN locations (e.g., database tables, log files), and remediation actions for unauthorized storage (e.g., deletion, encryption).
Change Impact Assessments 12.5.2, A3.2.2 (DESV) Documentation of scope impact assessments for significant system/network changes (e.g., new servers), confirming scope accuracy post-change.

Example: A change management ticket (e.g., from ServiceNow) for a new payment application, including a checklist confirming scope review, updated data-flow diagrams, and PCI DSS control verification.
Wireless Environment Assessment 11.2.1 Evidence of rogue wireless detection scans (even if wireless is not used) and, if wireless is used in the CDE, configurations showing secure settings (e.g., WPA3, changed default keys).
Example: A quarterly wireless scan report from a tool (e.g., AirMagnet) showing no unauthorized access points, plus AP configuration screenshots for CDE wireless networks showing secure encryption.
Unintended Data Receipt Handling 12.5.2 Procedures and records for handling CHD/SAD received via unintended channels (e.g., email), showing inclusion in the CDE or secure deletion with preventive measures. Example: A procedure document outlining steps to delete CHD from email servers and block future receipt (e.g., DLP rules), plus an incident log showing deletion of unsolicited PAN with email filter updates.
