The Security Scanning domain ensures that systems and networks in the Cardholder Data Environment (CDE) are regularly tested to identify and remediate security vulnerabilities. The evidence below is consolidated for functional stakeholders (e.g., IT, security, network teams), using practical, real-world artifacts aligned with PCI DSS v4.0.1 requirements.

Evidence Requirement PCI DSS Requirement(s) Description and Practical Examples
Security Testing Policy 11.1.1 A documented policy defining processes for regular security testing of CDE systems and networks, including vulnerability scanning and penetration testing. Example: A PDF policy document titled "Security Testing Policy" outlining quarterly vulnerability scanning and annual penetration testing procedures for CDE systems, with defined roles for IT and security teams.
Internal Vulnerability Scan Reports 11.3.1, 11.3.2 Quarterly internal vulnerability scan reports for CDE systems, conducted using authenticated scans, with identified vulnerabilities risk-ranked and remediated. Example: A Tenable Nessus scan report in PDF listing CDE server vulnerabilities (e.g., outdated software), with CVSS scores and a follow-up report showing remediation (e.g., patched software versions).
External Vulnerability Scan Reports 11.3.3, 11.3.4 Quarterly external vulnerability scan reports conducted by an Approved Scanning Vendor (ASV), with passing results for the last 12 months and evidence of resolved vulnerabilities. Example: An ASV scan report from Qualys in PDF showing a passing result for CDE external IPs, plus rescan reports confirming remediation of initially identified vulnerabilities (e.g., weak TLS versions).
Penetration Testing Reports 11.4.1, 11.4.2, 11.4.3, 11.4.4 Annual internal and external penetration test reports for CDE systems and networks, conducted by qualified personnel, with defined scope, real-world attack simulations, and remediated vulnerabilities. Example: A penetration test report in PDF from a QSA or third-party (e.g., Rapid7) detailing tested CDE systems, exploited vulnerabilities (e.g., SQL injection), and evidence of fixes (e.g., updated application code).
Segmentation Penetration Testing Records (DESV) A3.2.3, 11.4.5 For Designated Entities using segmentation, penetration test results conducted every 6 months (and after segmentation changes) to validate segmentation controls isolating the CDE. Example: A semi-annual penetration test report in PDF confirming no access from non-CDE to CDE systems, with firewall ACLs and test results verifying segmentation effectiveness.
Intrusion Detection/Prevention Configurations 11.5.1, 11.5.2 Configurations of intrusion-detection or prevention systems (IDS/IPS) monitoring CDE network traffic, with evidence of alert generation and response to suspicious activities. Example: Screenshots from an IDS/IPS console (e.g., Cisco Secure IPS) showing active monitoring of CDE traffic, plus a log of alerts (e.g., detected port scans) with response actions documented in a ServiceNow ticket.
File Integrity Monitoring Configurations 11.6.1 Configurations of file integrity monitoring (FIM) tools on CDE systems to detect changes to critical files, with evidence of alerts for unauthorized changes. Example: A Tripwire or OSSEC configuration screenshot showing FIM enabled on CDE server files (e.g., payment application configs), plus an alert log documenting detected file changes and investigations.
