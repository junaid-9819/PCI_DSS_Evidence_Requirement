The Access Control domain ensures that access to system components and cardholder data in the Cardholder Data Environment (CDE) is restricted based on business need-to-know and authenticated appropriately. The evidence below is consolidated for functional stakeholders (e.g., IT, security, system administration teams), using practical, real-world artifacts aligned with PCI DSS v4.0.1 requirements.

Evidence Requirement PCI DSS Requirement(s) Description and Practical Examples
Access Control Policy 7.1.1, 7.2.1, 7.2.2 A documented policy defining access control processes for CDE systems and data, based on least privilege and business need-to-know, with role-based access assignments. Example: A PDF policy document titled "Access Control Policy" outlining role-based access for CDE systems (e.g., payment admins vs. support staff), with procedures for granting and revoking access.
User Account Configurations 8.1.1, 8.2.1, 8.2.2 Configurations showing unique user IDs for all CDE system accounts, with no shared or generic accounts, and evidence of disabled vendor-supplied default accounts. Example: Active Directory (AD) screenshots showing unique user IDs (e.g., jdoe@company.com) for CDE server access, with disabled default accounts (e.g., "Administrator" renamed or disabled).
Authentication Procedures 8.1.2, 8.3.1, 8.3.2, 8.4.1, 8.4.2, 8.4.3 Documented procedures and configurations for secure authentication, including strong password requirements (e.g., minimum length, complexity) and multi-factor authentication (MFA) for remote and non-console CDE access. Example: An SOP detailing password policies (e.g., 12 characters, mixed case) and MFA setup, plus Okta or Duo configuration screenshots showing MFA enabled for VPN and CDE server access.
Access Control Lists 7.2.3, 7.2.4 Lists or configurations defining access permissions for CDE systems and data, reflecting least privilege and role-based access. Example: An Excel spreadsheet or AD group policy report listing CDE roles (e.g., "Payment DB Admins") and their permissions (e.g., read-only for reporting, full access for admins), verified by system access logs.
User Access Review Records 7.2.5, A3.2.5 (DESV) Records of quarterly reviews of user access privileges to CDE systems, ensuring alignment with business needs, with remediation actions for unauthorized access. Example: A quarterly access review report in PDF summarizing AD account audits, noting removed or adjusted accounts (e.g., ex-employee access revoked), with management sign-off.
Account Termination Procedures 8.2.6 Documented processes and evidence for timely termination of CDE access for terminated personnel or inactive accounts (e.g., within 90 days). Example: An HR-IT procedure in Word for disabling accounts upon termination, plus a ServiceNow ticket log showing a terminated employee's CDE access revoked within 24 hours.
Privileged Account Management 8.2.3, 8.2.4, 8.2.5 Evidence of controls for managing privileged accounts (e.g., admin accounts) in the CDE, including restricted access and monitoring. Example: A CyberArk or BeyondTrust report showing privileged account usage on CDE servers, with access restricted to specific admins and session monitoring logs.
